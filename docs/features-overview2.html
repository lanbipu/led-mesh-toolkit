管理项目工作区

project workspace · recents · examples

进入应用先看到的「门口」。从示例一键开一个新项目，或者打开磁盘上 已经存在的项目继续干活。最近项目按打开时间排序， 路径在写入时统一规整过，所以同一个项目不会因为绝对路径写法不同 在列表里出现两次。

  • 查看最近打开过的项目（名字 · 路径 · 上次打开时间）
  • 从示例建项目：curved-flat 8×4 平面屏，或 curved-arc 16×6 弧面屏
  • 打开已有项目文件夹（自动读 project.yaml）
  • 把项目从最近列表里移除（只是从列表删，不动磁盘文件）
cli hook
$ lmt project list-recent $ lmt project add-recent <abs_path> <display_name> $ lmt project load <abs_path> $ lmt project remove-recent <id> --yes

设计屏体形状

screen geometry · cabinet mask · references

告诉系统这块 LED 屏长什么样：cabinet 怎么排、有没有缺角、是 平面还是带弧度、坐标系原点定在哪个点上。这些设定决定了后面 重建怎么算、坐标怎么摆，保存后写回项目里的 project.yaml。

  • 排 cabinet 阵列：列 × 行、单个尺寸（mm）、每个 cabinet 多少像素
  • 遮掉缺失的 cabinet：拼异形屏、L 型、镂空都靠这一步
  • 选形状：flat 平面，或 curved 弧面（带半径）
  • 标三个参考点：原点 / X 轴方向点 / XY 平面点
  • 改动支持撤销重做，保存时原子写回 project.yaml
cli hook
$ lmt project load <abs_path> $ lmt project save <abs_path> --input - --yes

导入现场测量数据

total-station csv → measured.yaml

现场用全站仪打完点，会得到一堆 (name, x, y, z)。 丢给系统，它会自动认出每个点对应屏上哪个 cabinet 顶点，剔掉 明显打偏的点，没测到的点按形状先验补出来。最后产出一份 measured.yaml + 一张分类报告（实测 / 补出 / 离群 / 缺失）。

  • 导入全站仪 CSV，写出 measurements/measured.yaml
  • 同步生成 import_report.json（四类点各多少）
  • 手写好的 measured.yaml 也能直接加载，跳过这一步
  • CSV 里的点和屏 ID 不匹配会被挡住，避免误把别块屏的数据塞进来
cli hook
$ lmt total-station import <project> <screen_id> <csv> --yes $ lmt measurements load <path/to/measured.yaml>

生成施工指令卡

on-site instruction sheet

给现场测量员打一张「指引卡」：屏体方位图 + 三个参考点 + 全部 目标点编号。他们拿到这张纸到现场，就知道用全站仪先打哪几个、 按什么顺序走。命令行只输出 HTML，PDF 渲染走系统自带 WebView， 所以只能在 GUI 里导出。

  • 渲染指引卡 HTML：屏体方位 + 三个参考点 + 全部目标点编号
  • 目标点带顺序提示，便于按列 / 按行测
  • GUI 内一键导出 PDF；命令行只到 HTML，agent 用什么 renderer 自己挑
cli hook
$ lmt total-station instruction-card <project> <screen_id> # PDF 渲染只在 GUI 可用，命令行调用会返回 unsupported (exit 7)

几何重建：点 → mesh

surface reconstruction · quality metrics

项目最核心的一步。给它 measured.yaml + 屏体形状，它 把每个 cabinet 顶点的空间位置反算出来，连成三角网格，再算出 重建质量（RMS 误差、最大误差等）。结果写到磁盘的 JSON 报告， 同时在数据库里记下这一次运行，供下一项功能回查。

  • 输入：测量点 + cabinet 阵列 + 形状先验
  • 输出：mesh 顶点 + 三角面 + 质量指标 (QualityMetrics)
  • 报告落地：reports/2026-05-17-145532.json
  • 数据库记一行 reconstruction_runs（时间 · 屏 · RMS · 顶点数 · 导出目标）
cli hook
$ lmt reconstruct surface <project> <screen_id> measurements/measured.yaml --yes

查看运行历史

runs ledger · full json report

重建跑过的每一次都留底。这一块让用户回头看：哪天测的、用的 哪块屏、哪份测量集、RMS 多少、顶点几个、最后导出成了什么 OBJ。 点开任意一行还能拉出当时完整的 JSON 报告，方便比较前后两次的 差异。

  • 列出某个项目所有重建运行（可按屏 ID 过滤）
  • 每行带：时间 · 屏 · 方法 · RMS · 顶点数 · OBJ 输出
  • 展开一行，拿到完整 JSON 报告（mesh 结构 + 质量分布 + 参数）
cli hook
$ lmt reconstruct list-runs <project> --screen-id MAIN $ lmt reconstruct get-run-report <run_id>

导出给下游 VFX / 引擎

obj export · disguise · unreal · neutral

重建出来的 mesh 自己看没用，要送回演出现场或者游戏引擎才有 价值。这一块把某次重建结果导成 OBJ，按目标平台做坐标系转换、 单位转换、顶点焊接、三角化。同一次重建可以分别导成三种目标， 三份 OBJ 互不影响。

  • Disguise：VFX 演出系统（现场推屏）
  • Unreal：虚幻引擎（虚拟制作 / 游戏）
  • Neutral：中性坐标系，无特定目标平台
  • 可设：焊点容差 · 是否三角化 · 自定义文件名
cli hook
$ lmt export obj <run_id> disguise --dst /path/to/out.obj --yes $ lmt export obj <run_id> unreal --yes $ lmt export obj <run_id> neutral --yes
format.obj

每张卡上的标签什么意思

  • READ_ONLY · 只读，随便跑
  • WRITE_SAFE · 会写数据，但不需要 --yes
  • DESTRUCTIVE · 必须 --yes 或 --dry-run
  • CLI ✓ · 命令行能跑（headless）
  • GUI ✓ · 桌面应用里能点
